Основна ідея - CRM для ведення малого бізнесу. Зараз використовується Trello з автоматизаціями, але наразі його не достатньо.

Основні поінти:

1. Універсальність - це має бути абсолютно універсальне рішення, не заточене на конкретний бізнес.
2. Розширюваність - розширювання за допомогою динамічних модулів на JS.
3. Обєктно-оріжнтована структура - вся робота на основі колекцій. Дані зберігаються у **Cloud Firestore** (рішення оновлено з первісного "JSON у Firebase Storage", щоб отримати вбудовану офлайн-синхронізацію, запити та транзакції). Користувач входить через Firebase Auth (**лише Google**), а його дані лежать у його робочому просторі (`workspaces/{uid}/…`), захищеному security rules. Колекції та картки — типізовані документи Firestore (значення полів зберігаються нативними типами Firestore, дати як `Timestamp`), тож дані залишаються придатними для запитів і експортуються у відкритий JSON будь-коли. Це дозволяє ділитися даними між акаунтами та командами (шлях workspace вже сумісний з майбутнім спільним доступом; приватна папка на Google Drive такого не дозволяє). Синхронізація — вбудована у Firestore (на старті, безперервна, офлайн-черга записів); конфлікти вирішуються за принципом last-write-wins з видимим попередженням. Великі файли (поля типу "файл") вантажаться у Firebase Storage (пізніший епік).
4. Веб-інтерфейс - користувачі мають можливість взаємодіяти з системою через веб-інтерфейс.
5. Мобайл-версія - додаток має збиратись і на телефон.

Технічні аспекти:
1. Реалізація на Flutter
2. Серверна частина - дані користувача у Cloud Firestore (типізовані документи у його workspace, захищені security rules), автентифікація через Firebase Auth (лише Google). Адміністраторська частина також на Firebase: верифікація по пошті, адмін-доступ для певних юзерів, що дозволить вмикати та вимикати фічі для користувачів.
3. Універсальний інтерфейс - веб та мобайл.
4. UI/UX - має бути на основі Material Design та загальноприйнятих стандартів. Має виглядати сучасно, зручно, не просто generic кнопки та таблиці. Використовуй Claude design для проектування інтерфейсу.
5. Модульність - ядро (основна логіка системи, колекції з даними, їх менеджмент), апі (доступ до даних), маркетплейс з модулями на основі файрбейс та SPA (single-page application). Модулі на JS.
6. Картки в колекції мають різні поля з різними типами даних (текст, число, дата, файл, колекція тощо). Ці поля можуть бути калькульовані. Наприклад поле Вартість, яка бере значення з іншого поля та виконує математичні операції. Або не просто з іншого поля, а з іншої картки колекції (по айді).
